## C++中的读写效率

**效率对比梯队**<hr />

**垫底梯队：无优化的 cin / cout**

    ◦ 原因：C++ 为了兼容 C 语言，默认让 cin/cout 和 scanf/printf 保持同步（绑定同一个缓冲区）。每次输入输出都要处理复杂的 C++ 类型安全检查、locale（地域化）设置等。

    ◦ 适用场景：数据量 𝑁<10^4。

**中坚梯队：解绑后的 cin / cout**

    ◦ 做法：加上 ios::sync_with_stdio(false); cin.tie(0);

    ◦ 效率：提速 3~5 倍，接近 scanf/printf。

    ◦ 适用场景：绝大多数普通算法题（𝑁≈10^5∼10^6）。

**优秀梯队：scanf / printf**

    ◦ 原因：直接操作 C 语言的缓冲区，去掉了 C++ 的类型安全包装，通过格式化字符串（如 %d）直接解析。

    ◦ 适用场景：需要读取特定格式（如 %d:%d 读入时间），且数据量较大时。

**极速梯队：基础快读（getchar/putchar 版）**
◦ 效率：比 scanf/printf 快 2~3 倍。

    ◦ 原因：连 %d 格式字符串的解析开销都省了，直接一个字符一个字符读入内存，用位运算拼接。

    ◦ 适用场景：𝑁≈10^6∼10^7，或者被题目恶意卡常时。

**天花板梯队：究极快读（fread/fwrite 版）**

◦ 效率：比基础快读再快 2~5 倍，通常快到成为代码中的耗时 0。

◦ 原因：系统调用（让 CPU 去读硬盘/键盘）是极其慢的。getchar 读一个字符就可能触发一次系统调用；而 fread 是一次性把文件全部读到内存（缓冲区）里，后面的读取纯粹是内存里的指针移动，没有任何系统调用开销。

◦ 适用场景：极其恶劣的卡常题、字符串处理题（如 AC 自动机处理百万级文本）、或者读入 10^7 级别以上的大数据。
