# AGENTS.md

本文件用于说明后续编码代理在本仓库中的工作方式。

## 仓库概况

这是一个个人 C/C++ 算法仓库，主要内容包括算法模板、基础知识笔记和洛谷刷题记录。大多数 `.cpp` 或 `.c` 文件都是独立程序，拥有自己的 `main()` 函数。仓库目前没有统一的构建系统、包管理配置或共享运行时。

主要目录：

- `基础知识/`：C++ 基础、输入输出、STL 等 Markdown 笔记。
- `输入输出/`：快读快写模板和输入输出性能说明。
- `动态规划/`：动态规划模板，例如 01 背包、完全背包、多重背包。
- `数学/`：数学相关模板，包括高精度运算和数论筛法。
- `前缀和差分/`：前缀和、差分模板及相关说明。
- `洛谷刷题/`：按题型整理的洛谷题解，例如 `模拟/`、`字符串/`、`递归/`、`高精度/`、`数学/`。

## 工作规则

- 优先做小而可审查的改动；除非明确要求，不要大范围重构。
- 编辑前先确认要修改的文件，并用 3-6 条简短要点说明计划。
- 不要凭空编造路径、构建配置、辅助库或在线评测接口；不确定时先搜索仓库。
- 保持现有目录结构和命名风格，尤其是中文目录名和题目名。
- 默认用中文解释，内容要简洁、具体、可直接执行。
- 不要添加分析统计、遥测、网络请求、凭据或任何秘密信息。

## C/C++ 代码风格

- 除非用户要求抽成公共模板，否则每个题解文件保持自包含。
- 遵循本仓库的竞赛代码风格：
  - 新增题解时，文件顶部尽量保留题目链接或简短题意说明；
  - 输入量大或性能敏感时优先使用 `scanf/printf`；
  - 小中等输入规模可以使用 `cin/cout`，建议配合 `ios::sync_with_stdio(false); cin.tie(nullptr);`；
  - 仓库中常见 1 下标数组写法；
  - 可以使用中文注释，必要时优先使用中文说明算法意图。
- 简单在线评测题不要引入不必要的抽象。
- 对数组范围、循环边界和输入约束保持显式处理。
- 只在算法意图或边界条件不明显时添加注释，避免无意义注释。

## 修改现有题解

- 先完整阅读目标文件，理解当前算法和数据流。
- 修复 bug 时说明：
  - 可能原因；
  - 用什么最小实验或检查确认；
  - 最小修复方式。
- 除非确实需要，不要把一个已有题解整体重写。
- 避免无关格式化改动。
- 如果文件当前不完整或无法编译，只修复用户要求的问题；除非用户要求完整整理。

## 构建与验证

仓库没有统一测试命令。修改代码后，应优先编译被修改的单个源文件。

推荐 PowerShell 命令：

```powershell
g++ -std=c++17 -O2 -Wall -Wextra -o .\tmp_solution.exe "相对\路径\到\文件.cpp"
```

C 文件可使用：

```powershell
gcc -std=c11 -O2 -Wall -Wextra -o .\tmp_solution.exe "相对\路径\到\文件.c"
```

如果题目有样例输入，运行编译后的程序，并对照样例输出检查。

完成代码修改前：

- 如果本机可用 `g++` 或 `gcc`，运行最快的相关单文件编译检查；
- 汇报实际运行的命令和是否通过；
- 如果无法编译，明确说明原因。

临时构建产物，例如 `tmp_solution.exe`，不要提交到仓库。

## 文档更新

- 只有新增、移动或重命名重要模板/题解目录时，才更新 `README.md`。
- 保持相对链接风格，并与现有中文目录名一致。
- Markdown 笔记优先写清晰示例和短说明，不要写成过长的泛理论文章。

## Git 与安全

- 不要运行 `git reset --hard`、强制推送、递归删除等破坏性命令，除非用户明确要求。
- 工作区可能存在用户自己的改动。不要回滚不是你做的修改。
- 如果 Git 或网络命令卡住、要求输入凭据，停止并报告现象，不要反复重试。
- 不要打印或保存 token、密码、私钥、`.env` 值，或任何可能包含秘密信息的凭据输出。

## 回复格式

如果做了代码修改，最后说明：

- 简短总结；
- 修改的文件；
- 验证命令和结果。

如果是在调试问题，说明：

- 假设；
- 做过的实验；
- 最小修复或下一步具体操作。
