## 输入输出流

<hr />

### cin
``cin`` 是 C++ 标准库中最常用的输入流对象，属于 ``std::istream`` 类。它代表了标准输入（通常是键盘）。

<br />

**一、 基本读取特性**
1. **缓冲输入**：``cin`` 不是直接从键盘读取，而是从输入缓冲区读取。用户敲击键盘的内容会先存入缓冲区，当用户按下回车键后，``cin`` 才开始从缓冲区中提取数据。
2. **自动跳过空白字符**：使用提取运算符 ``>>`` 时，``cin`` 会默认忽略开头的所有空白字符（空格、制表符 ``\t``、换行符 ``\n``），直到遇到非空白字符才开始读取。
3. **以空白字符作为分隔符**：读取数字或单词时，一旦遇到空白字符，``cin`` 就会停止当前读取（但不会消耗掉这个空白字符，它仍留在缓冲区中）。
4. **链式调用**：因为 ``>>`` 运算符重载后返回的是 ``cin`` 对象本身的引用，所以支持 ``cin >> a >> b >> c;`` 这样的链式输入。

<br />

**二、类型安全与重载特性**
1. **类型安全**：不需要像 C 语言的 ``scanf`` 那样使用 ``%d``、``%s`` 等格式控制符。``cin`` 会根据变量的类型自动解析输入，避免了类型不匹配导致的内存越界错误。
2. **内置类型全面支持**：原生支持 ``int``, ``double``, ``char``, ``bool``, ``string`` 等所有基本类型。
3. **可扩展性**：可以通过重载 ``operator>>`` 让 ``cin`` 直接支持自定义的结构体或类。

<br />

**三、 状态管理机制（极其重要）**
``cin`` 内部维护了一个状态标志位，这对错误处理非常关键：

1. **正常状态**：一切顺利，可以继续读取。
2. **遇到文件结束符**：例如在命令行按下 ``Ctrl+Z``（Windows）或 ``Ctrl+D``（Linux/Mac）。
3. **格式错误/逻辑错误**：当期望读取数字却输入了字母时触发。此时 ``cin`` 会进入“死机”状态，并且这个错误的字符会留在缓冲区里，如果不处理，后续的所有 ``cin`` 都会失效。
4. **严重错误**：发生不可恢复的 I/O 错误（如磁盘损坏，键盘断连，极少发生）。
5. **布尔求值**：可以直接把 ``cin`` 放在 ``if`` 或 ``while`` 中判断，如 ``if (cin >> x)``，如果读取成功且未到达 ``EOF``，则为 ``true``。

<br />

**四、 性能与同步特性（竞赛与高性能场景必知）**
1. **与 C 标准流的同步**：默认情况下，C++ 为了保证可以和 C 语言的 ``scanf``/``printf`` 混合使用而不出错，``cin`` 会与 ``stdio`` 保持同步。这会带来不小的性能损耗。
2. **解除绑定提速**：通过添加 ``ios::sync_with_stdio(false); ``可以解除这种同步，大幅提升 ``cin`` 的速度（在算法竞赛中常与 ``cin.tie(0)`` 连用，使其速度媲美 ``scanf``）。注意：解除后禁止混用 ``scanf``/``printf``。

<br />

**五、 格式化控制特性**
通过与 ``<iomanip>`` 库配合，``cin`` 可以控制读取格式：

- ``cin >> hex >> a;``：以16进制读取整数。
- ``cin >> oct >> a;``：以8进制读取整数。
- ``cin >> ws;``：非常实用的操纵器，用于提取并丢弃缓冲区中所有的连续空白字符。

#### 💡 附：cin 的经典“踩坑”与解决方案
**坑 1：混合使用 ``>>`` 和 ``getline()`` 导致跳过输入**
```cpp
int age;
string name;
cin>>age;                   //假设输入20并回车
getline(cin,name);          //name直接变成了空字符串！
```
**原因**：``cin >> age`` 读取了 ``20``，但把回车符 ``\n`` 留在了缓冲区。紧接着 ``getline`` 看到 ``\n``，以为是一行结束，直接读取了一个空行。<br />
**解决**：
```cpp
cin >> age;
cin.ignore(); // 消耗掉遗留的 \n (或者用 cin >> ws)
getline(cin, name);
```

**坑 2：输入类型不匹配导致死循环**
```cpp
int x;
while (cin >> x) {
    cout << x << endl;
}
// 如果输入字母 'a'，程序会无限输出上一次的正确值或死循环。
```
**原因**：输入 ``'a'`` 时，``cin`` 进入 ``failbit`` 状态，且 ``'a'`` 留在缓冲区。下次循环 ``cin >> x`` 发现自己是 ``fail`` 状态，直接拒绝读取并返回 0（或 false），但不清理缓冲区，导致无限触发失败。<br />
**解决**：必须手动重置状态并清空缓冲区。
```cpp
int x;
while(true){
    if(cin>>x){
        cout<<x<<endl;
    }else{
        //清除错误状态
        cin.clear();
        //清空缓冲区里的垃圾字符（直到遇到换行符）
        cin.ignore(numeric_limits<streamsize>::max(),'\n');
        cout<<"输入有误，请重新输入数字！";
    }
}
```

**总结**：
``cin`` 的设计体现了 C++ **类型安全和面向对象**的思想，比 C 语言的 ``scanf`` 安全得多。但由于其**缓冲机制**和**严格的状态控制**，在处理混合数据类型（特别是数字后接字符串）时，必须对其内部缓冲区的残留字符保持警惕。

### cout
``cout`` 是 C++ 标准库中最常用的输出流对象，属于 ``std::ostream`` 类。它代表了标准输出（通常是控制台/屏幕）。与 ``cin`` 类似，``cout`` 在设计上充分体现了 C++ 的类型安全和面向对象特性，但在**缓冲机制**和**格式化控制**上有其独特之处。

<br />

**一、 基本输出特性**
1. **插入运算符 <<**：与 ``cin`` 的提取运算符方向相反，``<<`` 被重载为插入运算符，数据的流向是从变量到 ``cout``（即屏幕）。
2. **链式调用**：由于 ``<<`` 返回的是 ``cout`` 对象的引用，因此支持无限链式拼接：``cout << "A" << 10 << "B" << 3.14 << endl;``。
3. **连续输出无分隔**：与 ``cin`` 遇空格停止不同，``cout`` 只是纯粹地将数据推入缓冲区，不会自动添加空格。如果你写 ``cout << 1 << 2;``，屏幕上会显示 ``12``，需要手动加空格 ``cout << 1 << " " << 2;``。

<br />

**二、 类型安全与重载特性**
1. **无需格式控制符**：彻底告别 C 语言 ``printf`` 的 ``%d``, ``%f``, ``%s``，``cout`` 会自动根据变量的类型调用对应的重载函数，**杜绝了类型不匹配导致的未定义行为（如用 ``%f`` 打印 ``int``）**。
2. **自动识别指针**：如果你直接打印一个 ``char*``（C风格字符串），它会打印字符串内容；但如果你打印其他类型的指针（如 ``int*``），它会自动以十六进制打印内存地址（这是通过模板重载区分的）。
3. **支持自定义类型**：可以通过全局重载 ``operator<<``，让自定义类/结构体直接支持 ``cout << myObject;``。

<br />

**三、 缓冲与刷新机制（核心考点）**
``cout`` 不是一遇到 ``<<`` 就立刻把字符显示在屏幕上，而是先存放在**输出缓冲区**中。刷新（将缓冲区数据强制写到屏幕）的触发条件如下：

1. **缓冲区满了**：自动刷新。
2. **遇到显式的刷新指令**：如 ``endl``、``flush``。
3. **程序正常结束**：``main`` 函数返回时，标准库会自动清理并刷新。
4. **连接到终端时的行缓冲**：如果输出目标是控制台（而不是重定向到文件），当遇到换行符 ``\n`` 时，通常也会自动刷新。
5. **与 cin 的绑定（重要）**：默认情况下，``cout`` 和 ``cin`` 是绑定的（``tie`` 在一起）。这意味着**如果你正在用 ``cin`` 等待输入，系统会自动把 ``cout`` 缓冲区里的东西先刷到屏幕上**。这就解释了为什么 ``cout << "请输入:"; cin >> x;`` 不加 ``endl`` 也能看到提示语。

<br />

**四、 格式化控制特性**
cout 不使用格式字符串，而是使用操纵符。分为两种：

**1. 无参数操纵符（定义在 ``<iostream>`` 中）**：
- ``endl``：输出换行并**刷新缓冲区**。
- ``hex``、``oct``、``dec``：设置整数的十六进制、八进制、十进制输出（注意：这是“粘性”的，设置后一直有效，直到再次改变）。
- ``boolalpha`` / ``noboolalpha``：让 ``bool`` 值输出 ``true/false`` 而不是 ``1/0``。
- ``left`` / ``right``：设置对齐方式。
**2. 带参数操纵符（定义在 ``<iomanip>`` 中）**：
- ``setw(n)``：设置输出宽度（**非粘性，只对紧接着的下一次输出生效**）。
- ``setfill(c)``：设置填充字符（粘性）。
- ``setprecision(n)``：设置浮点数精度（粘性，通常配合 ``fixed`` 或 ``scientific`` 使用）。
- ``fixed``：强制以定点小数形式输出（如 ``3.140000``）。

<br />

**五、 性能与同步特性**
与 ``cin`` 完全一致，为了与 C 语言的 ``printf`` 保持兼容，默认处于同步状态。

1. **解除同步提速**：``ios::sync_with_stdio(false); ``可以解除与 ``stdio`` 的绑定，大幅提升 ``cout`` 速度。
2. **解除绑定提速**：``cin.tie(nullptr); ``可以解除 ``cin`` 对 ``cout`` 的自动刷新绑定，在频繁交替输入输出的算法题中能减少 overhead。


#### 💡 附：``cout`` 的经典“踩坑”与最佳实践
**坑 1：滥用 endl 导致程序卡顿**
```cpp
for (int i = 0; i < 1000000; ++i) {
    cout << i << endl; // 极慢！每次都强制刷新缓冲区，涉及系统级调用。
}
```
**解决**：在不需要立刻看到输出（如刷算法题、写文件）时，**坚决用 ``'\n'`` 代替 ``endl``**。
```cpp
for (int i = 0; i < 1000000; ++i) {
    cout << i << '\n'; // 极快！数据留在缓冲区，批量输出。
}
```


**坑 2：``setw`` 的“一次性”陷阱**
```cpp
cout << setw(10) << "Hello" << "World" << endl;
// 期望：Hello     World     (Hello占10格，World也占10格)
// 实际：Hello     World (只有Hello占10格，World紧贴其后)
```
**解决**：记住 ``setw`` 只对下一个元素生效。如果都要对齐，需要写两次：``cout << setw(10) << "Hello" << setw(10) << "World";``。


**坑 3：浮点数精度丢失**
```cpp
double pi = 3.141592653589793;
cout << pi << endl; // 默认输出 3.14159 （只显示6位有效数字，不是6位小数！）
```
**解决**：如果需要显示特定小数位数，必须配合 ``fixed``。
```cpp
cout << fixed << setprecision(10) << pi << endl; // 输出 3.1415926536
```

**总结**：
``cout`` 的核心难点就在于**理解并驾驭它的缓冲刷新机制和格式化状态（粘性与非粘性操纵符）**。在现代 C++ 开发中，``cout`` 是绝对的主流，但在极限性能要求的场景（如高频交易核心逻辑、超大规模数据刷屏），开发者有时仍会退回到 ``printf`` 甚至自行封装无锁缓冲区。

### 输入输出同步的开关

一般规则很简单：<br />

- `ios::sync_with_stdio(false); cin.tie(0);` 主要用于 加速 `cin/cout`
- 如果你只用 `cin/cout`，通常可以 关闭
- 如果你要混用 `scanf/printf` 和 `cin/cout`，通常应该 开启，或者干脆别混用

<hr />

#### 什么时候关闭？

适合这些情况：<br />

- 题目数据量大，担心 `cin/cout` 太慢
- 全程只用 C++ 风格输入输出：`cin`、`cout`
- 竞赛/刷题里需要更快的 IO 操作

常见写法：

```cpp
ios::sync_with_stdio(false);
cin.tie(0);
```

意思是：<br />

- `ios::sync_with_stdio(false);`：取消 C 和 C++ 标准流同步，提高速度
- `cin.tie(0);`：解除 cin 和 cout 的绑定，避免每次输入前都自动刷新输出

<hr />

#### 什么时候开启？

适合这些情况：<br />

- 你要混用 `printf/scanf` 和 `cout/cin`
- 你更在意输出顺序稳定，而不是极限性能
- 写交互题但你不太熟悉刷新机制时，要更谨慎

默认其实就是“开启同步”的，所以如果要开启，通常什么都不用写；或者写：

```cpp
ios::sync_with_stdio(true);
```

<br />

**重点结论:**

1. 只用 `cin/cout`：建议关闭
2. 只用 `scanf/printf`：不用管它
3. 混用两套 IO：建议开启同步，或者不要混用
4. 交互题：可以关闭，但要手动刷新，比如 `cout << x << endl;` 或 `cout << x << flush;`

<br />

#### 为什么混用时不要关闭？

因为关闭同步后，`cout` 和 `printf` 各自有自己的缓冲区，输出先后顺序可能和你写代码的顺序不一致。

例如：

```cpp
ios::sync_with_stdio(false);
cout << "A";
printf("B");
```

你可能以为输出是：

```cpp
AB
```

但实际顺序可能乱掉。
